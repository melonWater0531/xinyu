# UI 优化计划

> 适用项目：recamera_multimodal
> 适用页面：`/home`、`/v2`、`/device.html`
> 更新时间：2026-06-19
> 当前阶段：规划稿，尚未进入 UI 代码改造

## 1. UI Brief / 设计上下文

当前 UI 分成三类不同性质的页面：

1. `/home`：用户侧真实产品页，是“心屿”软件产品的初级形态。它需要承载核心功能入口、轻量控制、真实状态反馈和用户交互，包括情绪、专注度监测、多人场景跟踪、情绪日记、LLM 对话等。
2. `/v2`：开发/调试侧实时控制台，面向开发者和演示操作者，重点观察视频、视觉追踪、DOA、云台、日志、API 状态。
3. `/device.html`：旧版设备调试页，仅作为 legacy debug console 保留，不承担主流程展示。

因此 `/home` 不是产品介绍页，也不是纯展示页；它应该是用户真正会使用的产品主界面。更合适的方向是：

> `/home` 像一个温暖、可操作、有反馈的家庭陪伴产品；`/v2` 像一个清楚、可信、状态完整的开发调试控制台。

核心原则：

- `/home` 必须有所有用户侧核心功能的入口，并能完成真实功能或真实控制，而不是只做产品介绍。
- `/home` 的控制应是产品化控制：开始监测、查看状态、进入日记、与 LLM 对话、开启多人场景跟踪等；不要暴露开发调试噪声。
- `/home` 内容要丰富，但层级清楚，能适配手机小程序竖屏尺寸，并兼容 iPad 竖屏与横屏。
- `/v2` 承担高级调试、实时细节观察和工程控制，功能实现与展示要足够清楚。
- 多人模式下的 DOA 流程要说清楚：当前重点是声源方向辅助跟随，不是录音资料采集。

## 2. 当前页面诊断

### 2.1 `/home`

优势：

- 视觉气质比较完整，有“心屿”式的温暖陪伴感。
- 卡片、岛屿、底部导航已经形成平板端产品雏形。

问题：

- 产品能力、真实功能入口和调试信息混在一起，导致 `/home` 既不像完整产品，也不像清晰控制台。
- 当前内容还可以更丰富：应更系统地承载情绪、专注度、多人跟踪、情绪日记、LLM 对话、健康洞察等真实功能。
- 功能卡片不能只是介绍性卡片，要有明确入口、当前状态、下一步操作和结果反馈。
- 手机小程序尺寸下的信息密度、首屏节奏、底部导航需要重新组织，避免像桌面页面被简单压窄。
- iPad 竖屏与横屏应分别设计布局策略，而不是只依赖浏览器默认换行。

### 2.2 `/v2`

优势：

- 是目前最接近真实操作台的页面。
- 视频区域、单人追踪、多人 DOA、云台、安全控制、日志都已经具备。
- 适合作为后续主控制台继续优化。

问题：

- 信息密度偏高，视频、指标、按钮、日志之间缺少层级。
- 单人和多人模式的文案还带有“记录 / 保存记录”的残留，容易让人误解为仍在做音视频采集。
- 日志区域太抢眼，适合调试，但不适合演示时长期暴露在主视线中。
- dry-run、真实设备、WebSocket 断开、DOA 过期等状态缺少统一的视觉语言。

### 2.3 `/device.html`

优势：

- 保留了较多底层设备控制项。
- 可以作为 legacy debug console 参考。

问题：

- 视觉语言明显旧于 `/home` 和 `/v2`。
- 局部按钮和文字存在编码或标签损坏痕迹。
- 与 `/v2` 功能重叠较多，继续并列维护会增加理解和维护成本。

建议：后续将 `/device.html` 降级为 legacy 页面。`/home` 作为用户侧产品主界面独立存在，`/v2` 作为开发调试控制台独立存在。

## 3. 信息架构目标

建议将 UI 分成三类清晰页面：

```text
/home
  用户侧产品主界面
  负责：情绪监测、专注度监测、多人场景跟踪、情绪日记、LLM 对话、健康洞察、设备连接状态、产品化控制入口
  不负责：开发日志、底层 API 调试、裸云台参数调节

/v2
  开发/调试控制台
  负责：视频、单人追踪、多人 DOA、云台、安全控制、日志、高级状态诊断

/device.html
  legacy debug console
  负责：保留少量底层控制，不作为主流程入口
```

页面关系：

```text
/home
  -> 用户打开后即可使用核心产品功能
  -> 能启动/暂停情绪与专注监测
  -> 能进入多人场景跟踪
  -> 能查看情绪日记并与 LLM 对话
  -> 能看到设备连接、监测中、异常、离线等产品化状态

/v2
  -> 开发者真实执行单人视觉跟随、多人 DOA 跟随、云台控制和 debug

/device.html
  -> 仅在需要旧版底层调试时使用
```

## 4. 分阶段优化计划

### Phase 0：文案和状态真实性修正

目标：不大改结构，先把最容易误导用户的地方修掉。

任务：

- 将多人模式按钮从“开始记录 / 停止记录 / 保存记录”改为更准确的 DOA 语义：
  - “开始 DOA 跟随”
  - “停止 DOA 跟随”
  - “保存当前状态快照”或直接移除保存按钮
- 明确标识多人模式当前策略：
  - ReSpeaker/DOA 生产默认通过 WSL 内 USB control 输入，TCP 9999 仅备用。
  - ReSpeaker 已通过 `usbipd` 挂载到 WSL，DOA 读取与实体 LED 写入共用线程锁。
  - 声源跟随不默认保存音频；会议录音使用同一设备的 USB Audio Class 接口。
- 统一设备 IP 展示为 `RECAMERA_DEVICE_IP`。
- 在 `/home` 展示产品化运行状态，在 `/v2` 展示工程化运行状态：
  - `/home`：设备在线、监测中、暂停、数据过期、需要连接、权限不足、建议可生成
  - `/v2`：设备在线 / 离线、WebSocket 已连接 / 降级 HTTP 轮询、dry-run / real-control、DOA 新鲜 / DOA 过期 / 无 DOA 输入

验收：

- 演示者不需要看日志，也能判断当前系统是否可用。
- 普通用户在 `/home` 不需要理解 API 或调试术语，也能完成主要功能。
- 多人模式不会再让人误以为是在录制视频或音频资料。

### Phase 1：建立轻量设计 token

目标：让 `/home`、`/v2`、`/device.html` 不再像三个不同项目。

建议先不引入大型前端框架，继续保持当前 vanilla HTML/CSS/JS 的部署简单性。

建议 token：

```css
:root {
  --bg-page: #f6efe6;
  --bg-surface: rgba(255, 255, 255, 0.82);
  --bg-surface-strong: #fffaf3;
  --border-soft: rgba(78, 56, 38, 0.14);

  --text-primary: #2f261f;
  --text-secondary: #6f5c4b;
  --text-muted: #9a8776;

  --accent: #c7793f;
  --accent-soft: #f3d3b5;
  --success: #3f8f68;
  --warning: #d58b28;
  --danger: #c84a3a;

  --radius-sm: 10px;
  --radius-md: 16px;
  --radius-lg: 24px;

  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-6: 24px;
  --space-8: 32px;

  --focus-ring: 0 0 0 3px rgba(199, 121, 63, 0.28);
}
```

落地方式：

- 短期：在两个主页面各自 CSS 中统一变量名。
- 中期：抽出 `dashboard/ui_tokens.css`，由 `/home`、`/v2`、`/device.html` 共享。
- 暂不建议立刻迁移 React/Vue，避免把设备调试问题变成构建链问题。

### Phase 2：重构 `/v2` 为主控制台

目标：把 `/v2` 从“调试信息拼盘”优化为“实时控制台”。

推荐布局：

```text
┌──────────────────────────────────────────────┐
│ 顶部状态栏：设备 / 网络 / 模式 / 安全状态       │
├──────────────────────────────┬───────────────┤
│                              │ 模式控制       │
│ 视频主画面                    │ 单人追踪卡     │
│                              │ 多人 DOA 卡    │
│ overlay: bbox / center / doa │ 云台控制       │
│                              │ 安全停止       │
├──────────────────────────────┴───────────────┤
│ 可折叠日志 / 最近事件 / API 调试               │
└──────────────────────────────────────────────┘
```

关键调整：

- 视频区域成为视觉锚点，占据页面主要空间。
- 右侧只保留当前模式最相关的控制项。
- 非当前模式内容折叠或弱化，而不是同等权重常驻。
- 日志默认折叠，出现错误时自动提示。
- 危险操作单独放入 Safety 区域，不与普通按钮混排。

必须覆盖的状态：

- idle：设备在线但未启动任务。
- single-active：单人追踪中。
- multi-active：多人 DOA 跟随中。
- no-video：视频流不可用。
- no-doa：多人模式未收到 DOA。
- stale-doa：DOA 超过有效时间。
- dry-run：模拟控制。
- real-control：真实云台控制，必须有醒目标识。
- error：API 或 WebSocket 异常。

组件建议：

- `ConnectionBanner`：连接状态、设备 IP、WebSocket/HTTP 状态。
- `ModeSwitch`：单人 / 多人 / 停止。
- `VideoStage`：视频、追踪框、中心线、DOA 指示。
- `SingleTrackingCard`：目标数量、目标中心、跟随状态。
- `MultiDOACard`：角度、置信度、新鲜度、声源状态。
- `GimbalControlCard`：pan/tilt/速度/回中。
- `SafetyStrip`：停止跟随、停止云台、回安全位。
- `EventLogDrawer`：默认折叠，只在调试时展开。

### Phase 3：重构 `/home` 为用户侧产品主界面

目标：保留温暖气质，同时把 `/home` 明确做成可以真实使用的产品初级形态。用户应能在这里进入、控制或完成所有核心功能，而不是只看到产品介绍。

建议结构：

```text
Home dashboard：
  - 产品名、当前问候、设备连接状态
  - 今日情绪概览
  - 当前专注状态
  - 当前场景：单人 / 多人 / 未检测
  - 继续对话 / 开始监测 / 查看日记等主操作

Feature entry cards：
  - 情绪监测：开始/暂停、当前情绪、置信度、最近一次更新时间
  - 专注度监测：开始/暂停、专注时长、注意力趋势、提醒策略
  - 多人场景跟踪：开始/停止、当前说话方向、人数/声源状态、跟随状态
  - 情绪日记：今日记录、历史记录、生成总结、进入 LLM 对话
  - LLM 陪伴对话：围绕情绪日记、今日状态、专注情况进行问答
  - 健康建议：根据情绪、专注、互动状态生成轻量建议

Live feedback：
  - 监测中 / 暂停 / 设备离线 / 数据过期 / 权限不足
  - 多人 DOA 新鲜度
  - 情绪/专注数据来源说明
  - 隐私和本地处理提示

Official demo guide：
  - USB 初始化
  - 查看 wlan0 IP
  - 拔掉 USB 后浏览器访问官方 demo

Settings / about：
  - 设备 IP 与连接方式
  - 隐私与本地处理说明
  - 版本信息 / 开发调试入口
```

调整建议：

- `/home` 必须有真实功能入口，不能只放介绍性文案。
- 首屏重点是“现在能做什么”：继续对话、开始监测、查看日记、进入多人场景。
- 每个功能卡都必须同时回答四个问题：
  - 这个功能是什么？
  - 当前状态是什么？
  - 用户下一步能做什么？
  - 操作后会看到什么结果？
- 控制要产品化，不要暴露裸 API、底层日志或复杂云台参数。
- 单人/多人能力卡片使用“功能入口 + 状态 + 操作”结构：
  - 情绪监测：`开始监测` / `暂停监测` / `查看今日情绪`
  - 专注度监测：`开始专注记录` / `查看趋势` / `生成建议`
  - 多人场景跟踪：`开始多人跟踪` / `停止跟踪` / `查看声源方向`
  - 情绪日记：`写入今日记录` / `生成总结` / `和心屿聊聊`
- 官方 demo 打开方式可作为固定说明卡：
  1. 接 USB 初始化，Ubuntu 中 `ssh recamera@192.168.42.1`。
  2. 进入后执行 `ip addr`，查看 `wlan0` 无线 IP。
  3. 拔掉 USB，在浏览器输入最新 IP 打开官方 demo。
  4. 如需密码，统一使用 `IntelligentHardware0526_`。
- 官方 demo 说明卡可以保留在设置或帮助区，不要抢占主功能区。
- 情绪陪伴内容应该和真实日记、LLM 对话、建议生成连接起来。

#### `/home` 横竖屏与小程序适配矩阵

| 场景 | 布局策略 | 内容密度 | 导航策略 | 重点验证 |
|---|---|---|---|---|
| 手机小程序竖屏，约 375×667 到 430×932 | 单列产品工作台；首屏包含问候、状态、2-3 个最高频操作 | 高内容但分步展开，卡片先给状态和主按钮，详情二级展开 | 底部 tab：首页 / 监测 / 日记 / 对话 / 我的 | 单手触控可完成开始监测、查看日记、继续对话 |
| iPad 竖屏，约 768×1024 | 上方总览，下方 2 列功能卡；日记和对话可以上下分区 | 中高密度，允许展示趋势、状态和建议摘要 | 底部导航或左侧轻导航，触控尺寸优先 | 不出现桌面压缩感，功能入口不用横向滚动 |
| iPad 横屏，约 1024×768 到 1366×1024 | 左侧导航 + 中央总览 + 右侧日记/对话侧栏 | 高密度展示，但保留清晰主次和实时反馈区 | 左侧导航更合适，可保留顶部设备状态 | 横屏能同时查看监测状态、功能卡和 LLM/日记上下文 |

#### `/home` 功能完整度要求

必须具备：

- 首页总览：设备状态、今日情绪、专注状态、当前场景、最近日记、继续对话。
- 情绪监测：启动/暂停、状态展示、情绪结果、置信度或可信提示、更新时间。
- 专注度监测：启动/暂停、专注时长、分心提示、趋势摘要。
- 多人场景跟踪：启动/停止、声源方向、多人状态、跟随状态、DOA 新鲜度。
- 情绪日记：查看今日记录、查看历史、生成总结、手动补充、进入对话。
- LLM 对话：围绕情绪日记和今日状态追问、总结、建议；需要展示上下文来源。
- 健康建议：根据情绪、专注、互动生成建议，并能解释建议来自哪些观察。
- 设置/帮助：设备连接、官方 demo 打开方式、隐私说明、版本信息。

不建议出现：

- 裸调试日志常驻。
- 裸 API 状态表常驻。
- 复杂云台参数常驻。
- 只有介绍文字、没有可执行操作的“假功能卡”。
- 点击后没有状态变化、没有反馈、没有结果页的死入口。

### Phase 4：处理 `/device.html`

推荐策略：

- 不再作为默认入口。
- 页面顶部增加 legacy 提示：
  - “这是旧版设备调试页，推荐使用 `/v2`。”
- 修复损坏文案和按钮标签。
- 删除与 `/v2` 完全重复且无额外价值的入口。

如果后续时间紧张，可以先不改功能，只做三件事：

1. 修复乱码/损坏标签。
2. 加一个跳转 `/v2` 的主按钮。
3. 在 SOP 中说明该页仅用于 legacy debug。

## 5. 交互与可访问性要求

基础规则：

- 所有可点击元素触控尺寸不小于 `44px`。
- 所有按钮必须有明确 disabled、loading、error 状态。
- 不能只用颜色表达状态，必须配合文字或图标。
- 键盘可以完成基础流程：进入页面、切换模式、停止任务。
- 焦点态必须可见，建议使用 `--focus-ring`。
- 连接状态变化使用 `aria-live="polite"`。
- 错误和危险状态使用更明确的语义文本，避免只写“失败”。
- 支持 `prefers-reduced-motion`，减少动画和背景漂移。

危险操作规则：

- “停止跟随 / 停止云台 / 回安全位”永远可见。
- 真实云台控制模式需要醒目标识。
- dry-run 与 real-control 不能只靠日志区判断。

## 7. 验收清单

视觉与布局：

- `/home` 在手机小程序竖屏尺寸下，首页总览、核心功能入口和底部导航都可用。
- `/home` 在 iPad 竖屏下，功能卡、日记和对话入口不用横向滚动也能顺畅使用。
- `/home` 在 iPad 横屏下，可以同时呈现状态总览、功能入口和日记/LLM 侧栏。
- `/v2` 在 1366×768 桌面尺寸下，视频是主视觉锚点。
- `/v2` 在 iPad 横屏下，右侧控制区不挤压主画面。
- `/v2` 在 iPad 竖屏下，视频、模式控制、安全控制顺序清晰。
- `/v2` 在手机窄屏下，核心按钮不被底部导航或日志遮挡。

流程：

- 用户能在 `/home` 完成情绪监测的开始/暂停/查看结果。
- 用户能在 `/home` 完成专注度监测的开始/暂停/查看趋势。
- 用户能在 `/home` 开始/停止多人场景跟踪，并看到声源方向或 DOA 状态。
- 用户能在 `/home` 打开情绪日记、生成总结、手动补充记录。
- 用户能在 `/home` 基于情绪日记和当前状态与 LLM 对话。
- 用户能在 `/v2` 5 秒内判断设备是否在线。
- 用户能清楚区分单人视觉跟随与多人 DOA 跟随。
- 多人模式不再出现“旧采集资料 / 录音资料采集”的误导性表达。
- 用户不会把 `/home` 误解成静态产品介绍页。

状态：

- `/home` 的每个核心功能都具备 idle、loading、active、paused、empty、error、stale 状态。
- `/home` 的 LLM 对话需要展示上下文来源：情绪日记、今日状态、专注记录或用户手动输入。
- dry-run 与 real-control 明确可见。
- WebSocket 断开时有降级提示。
- DOA 无输入与 DOA 过期有不同提示。
- API 错误不会只出现在 console 或日志里。

可访问性：

- 只用键盘可以在 `/home` 完成主要功能入口访问、开始/暂停监测、打开日记、发送对话。
- 只用键盘可以在 `/v2` 完成进入控制台、切换模式、停止任务。
- 所有主按钮有可见 focus。
- 颜色对比满足基本可读性。
- 动画可被 reduced motion 降低。
- `/home` 的趋势、情绪卡片、日记摘要、LLM 对话不能依赖 hover 才能读到关键内容。

回归：

- `/api/state` 正常显示。
- `/api/tracking/start` 与 `/api/tracking/stop` 可用。
- `/api/multi/start` 与 `/api/multi/stop` 可用。
- `/api/gimbal/*` 控制项不被 UI 改造破坏。
- `/home` 的情绪、专注、多人跟踪、日记、LLM 对话入口要么接入真实接口，要么明确显示“待接入/模拟状态”，不能伪装成已实现。
- 日志折叠后仍可用于 debug。

## 8. 风险与取舍

- 暂不建议引入重型前端框架。当前系统的风险主要在设备链路、网络、DOA 和云台控制，UI 改造应保持部署简单。
- `/home` 应加强真实产品感和功能完整度，但不能塞入开发日志、裸 API 表格和复杂云台参数。
- `/v2` 应成为唯一开发调试控制台，避免 `/device.html` 与 `/v2` 长期并列演化。
- `/home` 中尚未真实接入的能力必须诚实标注模拟/待接入状态，避免把原型假装成可用功能。
- 多人模式的所有文案要围绕 DOA 与声源跟随，不再延续旧采集资料或录音资料的表达。

## 2026-06 Completion Notes

The control dashboard redesign has been implemented with a left sidebar and focused feature pages:

- Single scene: face tracking and analysis in one page.
- Multi scene: sound-source yaw follow and meeting recording.
- Sound-source / LED indicators are integrated into multi pages instead of a standalone page.
- Every page has `启动功能`; switching pages stops old control and never auto-starts the new page.
- The top device address input reconnects FastAPI video/perception only; real gimbal control remains owned by `main_phase3.py`.
- Buttons continue to call FastAPI APIs only, preserving the EventBus and single-control-plane architecture.
- Page activation now returns an authoritative `session_id`; controls unlock only after `main_phase3` accepts the session.
- A 1-second heartbeat renews a 2.5-second lease. Page loss, browser crash, or network loss automatically returns the runtime to inactive and stops the gimbal.
- Multi-person pages distinguish the physical ReSpeaker XVF3800 WS2812 ring from the UI direction diagram and show hardware/readback status.
- Gimbal yaw/pitch/speed values come from the Node-RED CAN readback bridge rather than frontend state.

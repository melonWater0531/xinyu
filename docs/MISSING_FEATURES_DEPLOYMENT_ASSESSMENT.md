## 2. 已部署功能复核

### 2.1 #4 主动情绪干预

**部署事实**：`core/emotion_intervention.py` 已在 FastAPI 感知循环初始化并持续更新，结果写入 `/api/state` 与 `/ws` 的 `proactive_intervention`。

**为什么能部署**：复用现有 `emotieff`、`attention`、`eye_metrics`、`gaze`，不增加重模型，不触碰云台控制。滑动窗口、置信度门槛和冷却时间都可在策略层独立调整。

**为什么值得部署**：它把系统从“展示状态”升级为“在持续状态下温和关心”，与心屿定位高度一致，并且能够为情绪关心 PWA 通知提供可靠入口。

**为什么仍需谨慎**：情绪分类不是心理诊断；真实环境中的误识别、采样频率和用户接受度尚需验收。文案必须保持非诊断、可关闭、低频率。

### 2.2 #8 手势识别陪伴交互 A 版

**部署事实**：`vision/gesture_detector.py`、FastAPI 调用、状态快照和 `/home` 五种 intent 处理均已接入。仓库当前缺少 `models/gesture_recognizer.task`，因此状态为“已部署（资源待补）”。

**为什么能部署**：使用 MediaPipe Gesture Recognizer，识别与 intent 映射集中在独立模块；连续 4 帧、0.6 置信度和 3 秒冷却限制误触发；模块只返回 UI intent。

**为什么值得部署**：张手、握拳、点赞、点踩和剪刀手都能对应低风险陪伴动作，演示清晰，也能积累轻量反馈。

**为什么不能宣称完成验证**：模型资源缺失；远距离、遮挡、低光和镜头视场会显著影响识别。补模型后仍需逐个手势做真实摄像头验收。

**控制边界**：不得发送 `gimbal_*`、`feature_*`、`dpad_*`；当前不做云台控制、自动录音或自动永久写日记。

### 2.3 #9 注视方向估计

**部署事实**：`vision/gaze_estimator.py` 使用 Face Landmarker 虹膜点输出粗粒度 gaze，并以 15% 权重融合到 `AttentionEngine`；状态通过 `/api/state` 与 `/ws` 输出。

**为什么能部署**：复用现有 MediaPipe 面部关键点，无新硬件、无外部服务，计算量相对小；缺失时原 attention 仍可运行。

**为什么值得部署**：补足“头朝前但眼睛离开屏幕”的情况，让专注和疲劳判断拥有更丰富的辅助证据。

**为什么仍需谨慎**：它不是精确眼动仪；眼镜、遮挡、眼睛大小、分辨率和光照都会影响结果。UI 只能表述为“视线趋势”。

### 2.4 #10 PWA 本地通知

**部署事实**：通知调度位于 `dashboard/home.html`，使用 Notification API、Service Worker 和 localStorage；`dashboard/sw.js` 处理通知点击。六类通知为护眼、久坐、喝水、疲劳、低专注和情绪关心。

**为什么能部署**：完全复用现有 PWA 和前端状态，不需要后端推送接口、账号体系或外部消息服务，控制链无侵入。

**为什么值得部署**：健康计时和实时状态终于能形成可达用户的提醒闭环，同时默认安静时段、同类冷却和无敏感正文降低打扰与隐私风险。

**为什么仍需谨慎**：Notification 和 Service Worker 需要 HTTPS 或 localhost；不同浏览器和移动系统对后台页面限制不同。第一版是页面驱动的本地提醒，不是服务器离线推送。

**范围澄清**：未接入 ntfy、Telegram、MQTT；不发送截图、日记正文、会议原文；不提供跌倒、异响或安防通知。

## 3. 未部署功能逐项判断

### 3.1 #1 TTS 语音输出

**可实现性**：高。可在 PC 侧接入 Piper、MeloTTS 或系统语音，并由前端/后端消费文本结果。

**部署理由**：能增强陪伴感和免手操作；可复用 LLM 与主动关心文本。

**不立即部署理由**：需要处理打断、音频设备抢占、会议录音回声、音量和隐私；当前明确不做 TTS。

### 3.2 #2 说话人分离与声纹-人脸绑定

**可实现性**：中。可引入 pyannote/声纹嵌入，并结合 DOA 和人脸轨迹做弱绑定。

**部署理由**：可提升会议记录可读性和多人场景理解。

**不立即部署理由**：模型重、同步复杂、错绑风险和生物特征隐私成本高。建议先用“左/中/右”或 Speaker A/B/C。

### 3.3 #3 跌倒/姿态异常检测

**可实现性**：高。现有 YOLO11 pose 可支持框宽高比、躯干角度、关键点高度和持续帧规则。

**部署理由**：技术投入低，适合照护类产品。

**不部署理由**：当前系统明确不走安防/告警路线；单摄像头误报可能把陪伴产品变成监控产品。本阶段标记“未部署（暂不计划）”。

### 3.4 #5 VLM 场景理解

**可实现性**：中-高。可复用 `/api/snapshot`，新增手动 `scene/ask` 接口调用本地或云端 VLM。

**部署理由**：能解释画面、补充日记和增强演示。

**不立即部署理由**：模型体积、延迟、显存/内存、幻觉和隐私成本高；不能用于实时控制或安全判断。

### 3.5 #6 自定义唤醒词与常驻语音助手

**可实现性**：中。`audio/wake_word.py` 已存在，但未进入主链路。

**部署理由**：能形成免手陪伴体验。

**不立即部署理由**：持续监听会和 DOA、录音争用音频设备，还需要 VAD、STT、LLM、TTS、打断和隐私提示的完整状态机。

### 3.6 #7 声音事件检测

**可实现性**：中-高。可用 YAMNet/AudioSet 在 ReSpeaker 音频窗上分类。

**部署理由**：可与 DOA 组合形成声源事件理解。

**不部署理由**：误报、CPU 开销、类别治理和安防叙事风险较高；当前明确不做异响检测与相关推送。

### 3.7 #11 Home Assistant / MQTT

**可实现性**：高。可将 `/api/state` 低频映射为 MQTT sensors，并增加 HA discovery。

**部署理由**：方便智能家居联动和外部仪表板展示。

**不部署理由**：增加 broker、鉴权、断线重连、主题治理和长期运维；当前计划明确排除 MQTT。

## 5. 当前优先级

1. 补齐 `models/gesture_recognizer.task`，对五个手势逐项验收。
2. 在真实浏览器/HTTPS 环境验证六类 PWA 通知、冷却和安静时段。
3. 用多光照、眼镜和不同距离样本校准 gaze 与 attention 融合。
4. 用录制状态序列验证主动情绪干预阈值、冷却和误触发。
5. 暂不开展跌倒、声音事件、MQTT、云台手势控制、ntfy/Telegram 和 TTS。

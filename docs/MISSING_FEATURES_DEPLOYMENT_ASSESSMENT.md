# 未实现功能可实现性与部署难度评估

> 本文档基于 `D:/Downloads/MISSING_FEATURES.md`、`docs/EXISTING_FEATURES.md` 和当前代码架构整理。目标是判断这些功能在现有架构上的可实现性、部署难度、部署理由和不部署理由。本文只做评估，不做具体部署。

## 1. 评估基准

当前架构已经具备几条很适合扩展的主干：

- 视觉主干：`recamera_fastapi.py` 接入 reCamera SSCMA 视频，已有 `pose_estimator`、`face_tracker_v2`、`mediapipe_face`、`emotieff_adapter`、`attention_engine`。
- 音频主干：ReSpeaker XVF3800 已提供 DOA/VAD 和 USB Audio，已有 `conversation_recorder`、`transcriber`、`network_doa`。
- 控制主干：真实云台控制统一收束在 `main_phase3.py -> Orchestrator -> FSM -> SafetyLayer -> RecameraClient`。
- 状态主干：WebSocket `/ws` 和 REST `/api/state` 已能承载新增状态字段。
- 产品主干：`/home` 已有情绪、专注、多人、日记、健康和 LLM 入口。

部署难度分级：

| 难度 | 含义 |
|---|---|
| 低 | 主要新增策略/轻量模块，复用现有数据，不引入重模型或复杂服务 |
| 中 | 需要新依赖、新 API、新前端状态或后台任务，但对主架构影响可控 |
| 高 | 需要大模型、异步长任务、模型下载/显存/许可证处理，或高误报风险治理 |
| 很高 | 需要改变交互主链路或引入持续常驻管线，稳定性和体验风险都较高 |

## 3. 单项评估

### 1. TTS 语音输出

**可实现性：高**

可以部署，因为现有系统已经有 `/api/chat`、`/api/reflect`、`/api/meeting/summarize` 这些文本输出点，只缺一个文本转语音出口。新增 `audio/tts.py` 和 `POST /api/speak` 不需要碰云台控制平面，也不影响 SafetyLayer。默认输出到 PC 扬声器，硬件边界清楚。

**部署难度：中**

主要难点不是架构，而是依赖和音频播放环境：Piper 模型下载、中文音色选择、Windows/WSL 下音频输出设备、播报队列、打断策略都需要处理。若直接从 WSL 播放到 Windows 扬声器，可能需要额外验证音频设备映射。

**为什么要部署**

- 补齐“唤醒/输入 -> LLM -> 语音回复”的陪伴闭环，产品感知价值很明显。
- 和主动情绪干预、跌倒询问、会议总结播报可以复用同一能力。
- 与现有控制链路解耦，风险集中在音频输出模块。

**为什么不要部署或暂缓**

- 如果当前演示主要通过网页完成，TTS 的收益不如跌倒/推送/注视这类可视化功能直接。
- 中文 TTS 音色质量和延迟会影响产品观感，粗糙播报反而削弱“心屿”体验。
- 需要明确播报隐私策略，会议内容、日记内容不应默认外放。

**建议方案**

先做手动触发的 `POST /api/speak` 和前端“朗读回复”开关，不要默认让所有 LLM 回复自动播报。等播放稳定后再接主动播报。

### 2. 说话人分离 + 声纹人脸绑定

**可实现性：中**

部分可部署。现有系统已有会议录音、Whisper 转写、DOA 和 ArcFace，人脸侧与音频侧都有基础。但“谁说了什么”不是简单拼接：说话人分离需要长音频时间轴，Whisper 需要句段时间戳，DOA 与画面中的人脸还要做时间同步和空间映射。

**部署难度：高**

难点包括 pyannote/diarize 模型依赖、CPU 速度、多人重叠说话、声纹聚类稳定性、DOA 与画面人的绑定逻辑、姓名管理 UI，以及模型许可证/API key 可能带来的部署约束。

**为什么要部署**

- 会议摘要从“内容摘要”升级到“归属化纪要”，价值很高。
- 已有 ReSpeaker DOA 和会议录音，方向信息可以作为低成本辅助。
- 对演示很有说服力：同一套硬件完成看、听、记、归属。

**为什么不要部署或暂缓**

- 完整声纹人脸绑定很容易出现错标，一旦错标“谁说了什么”，用户信任会明显下降。
- pyannote 类模型对环境、算力和依赖要求高，部署不如其他功能稳定。
- 需要保存更多音频/声纹/人脸特征，隐私风险上升。

**建议方案**

不要一开始做“真实姓名级绑定”。先做弱化版：

- 阶段 1：只用 DOA 区域标注“左侧发言人 / 正前方发言人 / 右侧发言人”。
- 阶段 2：加入 diarization，输出“Speaker A/B/C”。
- 阶段 3：用户手动确认后再绑定人脸或姓名。

### 3. 跌倒 / 姿态异常检测

**可实现性：高**

非常适合现有架构。`pose_estimator` 已输出 COCO-17 关键点和人体框，跌倒检测可以作为 `vision/fall_detector.py` 的纯规则模块接入 FastAPI 状态快照或 EventBus，不需要新模型，不需要新硬件。

**部署难度：低**

主要是规则调参和误报治理。算法可以先基于人体框宽高比、肩髋线角度、头/髋高度关系、持续帧数和速度变化做判断。

**为什么要部署**

- 独居陪伴/健康安全场景价值强，和“情绪陪伴”叙事兼容。
- 几乎不新增算力，不破坏主控制链路。
- 可以和推送通知、TTS 询问、自动抓拍形成完整事件闭环。

**为什么不要部署或谨慎部署**

- 单摄像头视角下，弯腰、躺沙发、坐下、瑜伽动作可能误报。
- 如果产品没有明确“安全告警”定位，跌倒检测可能让系统叙事从陪伴偏向监控。
- 真正告警不能只靠单帧或短时规则，必须加确认机制。

**建议方案**

优先部署，但只作为“姿态异常提醒”而不是医疗级跌倒检测。触发后先进入二次确认：TTS/前端询问 + 10-20 秒无响应再推送。

### 4. 主动情绪干预

**可实现性：高**

现有 EmotiEffLib、attention、LLM 对话和日记能力都已具备，新增一个策略层即可。它不需要修改云台控制，只需订阅状态快照或在 FastAPI 循环中维护滑动窗口。

**部署难度：低-中**

代码难度低，产品难度中。关键在触发频率、冷却时间、措辞和用户可关闭能力。

**为什么要部署**

- 让系统从“用户问才回答”变成“看见状态后主动关心”，非常贴合心屿定位。
- 无新模型，复用现有情绪流和 LLM。
- 可以和 TTS 组合，形成真正的陪伴感。

**为什么不要部署或谨慎部署**

- 情绪识别不是心理诊断，主动干预措辞如果过度判断会冒犯用户。
- 频繁打扰会显著降低体验。
- 负面情绪识别误判时，主动问候会显得突兀。

**建议方案**

可以部署，但使用保守规则：连续多分钟负面情绪 + 低专注/疲劳指标 + 冷却时间。文案避免诊断式表达，使用“要不要休息一下”这类轻提示。

### 5. VLM 场景理解

**可实现性：中-高**

架构上可实现。已有 `/api/snapshot` 可以提供 JPEG，新增 `vision/vlm.py` 和 `POST /api/scene/ask` 即可把图片和问题送入 Moondream/MiniCPM-V/Qwen-VL。与云台控制平面解耦。

**部署难度：高**

难点是模型体积、推理速度、显存/内存、量化环境、Prompt 稳定性，以及 VLM 对画面细节的误判。若使用 Ollama 或本地大模型，还要维护额外服务。

**为什么要部署**

- 能把系统从“检测数值”升级为“看懂画面”，对演示非常有吸引力。
- 可作为 `/api/chat` 的工具能力，让用户问“现在画面里有什么”。
- 对异常事件解释、日记补全、场景摘要都有增益。

**为什么不要部署或暂缓**

- 对当前核心闭环不是必需，重依赖可能拖慢系统。
- VLM 幻觉风险高，安全类判断不能依赖它。
- 如果 PC 算力一般，用户等待时间会破坏实时感。

**建议方案**

先做实验版，不接入实时循环，只做用户手动点击“分析当前画面”。不要让 VLM 驱动云台或安全告警。

### 6. 自定义唤醒词 + 常驻语音助手主链路

**可实现性：中**

现有 `wake_word.py`、ReSpeaker 音频和 Whisper/TTS 方向上可以支撑，但“常驻主链路”会改变系统交互方式。它需要持续监听、唤醒后录音、VAD 截断、STT、LLM、TTS、打断和状态管理。

**部署难度：很高**

难点不只是 openWakeWord 模型，而是全链路稳定性：误唤醒、漏唤醒、环境噪声、和会议录音抢音频设备、用户隐私提示、持续后台任务的资源占用。

**为什么要部署**

- 完成真正免手操作，是陪伴型设备的重要体验。
- 和 TTS 结合后，系统从 Web App 更像实体智能助手。
- 自定义“心屿”唤醒词有品牌感。

**为什么不要部署或暂缓**

- 当前音频链路已经承担 DOA 和会议录音，常驻助手会增加抢占和状态冲突。
- 误唤醒会让用户对隐私非常敏感。
- 自定义唤醒词训练和验证周期长，不适合快速稳定交付。

**建议方案**

暂缓主链路部署。先做按钮触发的“按住说话/点击对话”，再做固定内置唤醒词验证，最后再考虑自定义唤醒词。

### 7. 声音事件检测

**可实现性：中-高**

ReSpeaker 音频流和 DOA 已有，YAMNet 这类模型可以放 PC 跑滑窗分类。命中事件后可以复用 EventBus、推送、抓拍和 DOA yaw 方向。

**部署难度：中-高**

模型接入不算最难，难点在类别治理和误报。AudioSet 类别多，家庭环境声音复杂，敲门、呼救、玻璃破碎、哭声这类事件在真实环境中很容易混淆。

**为什么要部署**

- 和 DOA 天然互补：听到异响 -> 转向声源 -> 抓拍/通知。
- 扩展独居照护、安全提醒能力。
- 可独立运行，不需要改变 LLM 或云台主控制逻辑。

**为什么不要部署或谨慎部署**

- 误报告警会很烦，尤其是敲门/喊叫/哭声这类生活噪声。
- YAMNet 类别泛化不一定适合中文家庭场景。
- 持续音频分类会占用 CPU，并可能和录音/唤醒词争用音频流。

**建议方案**

只部署少量高价值类别，并用“声音事件 + DOA 有效 + 持续时间 + 时间段规则”联合触发。先进入通知/抓拍，不直接驱动强控制。

### 8. 手势识别陪伴交互

**可实现性：中**

可实现，但要看摄像头视角和距离。本轮直接采用 MediaPipe Gesture Recognizer 作为第二版识别框架，不再先做 YOLO-pose 规则版。手势第一阶段只作为免触碰陪伴交互入口，不控制云台。

**部署难度：中**

新增视觉模块不难，难在稳定识别、误触发防抖、前端反馈和日记/LLM 语义映射。由于 A 版不控制云台，也不外发告警，控制链路风险显著降低。

**为什么要部署**

- 演示效果强，用户不用碰网页即可和“心屿”互动。
- 适合做陪伴型快捷入口：唤起、静音、反馈、记录积极瞬间。
- 不进入云台控制链路，风险比“手势控制设备运动”低。
- 可以为后续个性化交互收集“用户喜欢/不喜欢哪类回复”的轻量反馈。

**为什么不要部署或谨慎部署**

- 远距离、低光、遮挡下手势识别不稳定。
- 即使不控制云台，误触发也可能造成错误记录日记或错误反馈 LLM 回复。
- 精细手势会额外增加视觉推理负担。

**建议方案**

先做 A 版低风险陪伴交互，所有手势只更新 `gesture.intent` 并触发前端轻反馈，不发 `gimbal_*` 或 `feature_*` 控制事件。

| 手势 | intent | 第一阶段功能 | 风险控制 |
|---|---|---|---|
| Open Palm / 张手 | `summon_xinyu` | 唤起心屿，显示“我在听”状态 | 只改变 UI 状态，不自动录音、不控制云台 |
| Closed Fist / 握拳 | `pause_or_mute` | 停止当前 TTS 播报或关闭当前主动提示 | 只影响本地播报/提示，不影响录音和云台 |
| Thumb Up / 点赞 | `feedback_positive` | 标记当前 LLM 回复或健康建议“有帮助” | 只写本地反馈状态，不立即调整模型 |
| Thumb Down / 点踩 | `feedback_negative` | 标记当前 LLM 回复或健康建议“无帮助” | 只写本地反馈状态，后续用于回复风格调优 |
| Victory / 剪刀手 | `capture_positive_moment` | 记录一个积极瞬间到日记候选 | 先生成待确认日记草稿，不自动永久保存 |

接口与状态建议：

- `vision/gesture_detector.py` 输出 `name`、`confidence`、`handedness`、`stable_frames`、`intent`、`intent_ready`。
- `/ws` 与 `/api/state` 新增 `gesture` 状态块。
- `/home` 展示手势轻反馈，如“已唤起心屿”“已标记有帮助”“已生成积极瞬间草稿”。
- `/control` 展示原始识别类别、置信度、稳定帧和映射后的 intent，方便调试。
- 稳定策略：同一手势连续 3-5 帧且置信度超过阈值才输出 `intent_ready=true`；同一 intent 设置 3 秒冷却。

### 9. 注视方向估计

**可实现性：高**

非常适合现有架构。已有 MediaPipe Face Landmarker 和 attention engine，只需开启/接入虹膜关键点或补充 gaze 几何估计，再把 gaze 特征融合进专注评分。

**部署难度：低-中**

主要是 MediaPipe 配置、关键点稳定性和个体校准。相比新增模型，风险小很多。

**为什么要部署**

- 明显提升专注评分质量：从“头朝向”升级到“眼睛是否看向屏幕/目标区域”。
- 不需要新硬件，可能也不需要新依赖。
- 前端可以展示“注视偏离”或“眼神疲劳”指标，产品解释性强。

**为什么不要部署或谨慎部署**

- 摄像头分辨率、佩戴眼镜、眯眼、光照会影响虹膜点稳定性。
- 如果没有校准，gaze 只能做粗略估计，不能宣称精确眼动追踪。
- 增加 MediaPipe refine 后可能带来一些推理延迟。

**建议方案**

建议部署。先把 gaze 作为 attention 的辅助证据，不直接替代原评分；前端文案用“视线趋势/可能偏离”，避免过度精确。

### 10. 推送通知

**可实现性：高**

非常容易接入。当前系统已有事件、状态、快照和后端 API，新增 `notify/` 模块即可对接 ntfy 或 Telegram。对主控制平面没有侵入。

**部署难度：低**

难点主要是配置管理：token、主题、用户开关、频率限制、隐私内容脱敏。

**为什么要部署**

- 让跌倒、异响、会议完成等事件离开浏览器，真正到达手机。
- 是很多安全/陪伴功能的闭环出口。
- 成本低，架构风险小。

**为什么不要部署或谨慎部署**

- 推送涉及外部服务和隐私，不能默认发送图片/日记/会议内容。
- 频繁推送会造成打扰。
- Telegram 在不同网络环境下可达性不稳定，ntfy 主题如果公开也需注意安全。

**建议方案**

建议部署，但默认只推送文字摘要，不默认附带快照。所有告警加冷却时间和用户开关。

### 11. Home Assistant / MQTT 集成

**可实现性：高**

现有 `/api/state` 和 WebSocket 已经有 presence、emotion、attention、DOA、gimbal、health 等状态，转成 MQTT sensor 很直接。HA discovery 也适合作为外部展示集成。

**部署难度：中**

主要是 MQTT broker、HA discovery topic、实体命名、状态频率限制、断线重连和配置项。

**为什么要部署**

- 能把系统接入智能家居自动化，比如低专注提醒、有人出现、异常声音联动灯光。
- 展示效果好，利于把项目从单 App 扩展成生态节点。
- 纯集成型，不需要新算法。

**为什么不要部署或暂缓**

- 如果当前没有 Home Assistant 展示需求，它对核心产品体验提升有限。
- MQTT 增加一个长期运行外部依赖，部署说明和排错成本会上升。
- 状态发布过频可能造成 broker 噪声，需要限流和主题设计。

**建议方案**

按展示需求部署。优先发布低频状态：presence、emotion、attention_score、doa_deg、fall_alert、sound_event。不要把每 200ms WebSocket 全量镜像到 MQTT。

## 4. 推荐部署组合

### 4.1 最稳妥组合：健康陪伴闭环

包含：

- #3 跌倒 / 姿态异常检测
- #9 注视方向估计
- #4 主动情绪干预
- #10 推送通知

理由：

- 都高度复用当前视觉/情绪/状态链路。
- 不引入重模型，不改变云台控制主干。
- 能形成“发现异常 -> 温和询问 -> 必要时通知”的完整闭环。

不建议同时加入：

- #6 常驻唤醒词，因为会扩大音频链路复杂度。
- #2 完整说话人绑定，因为会把第一阶段目标拖入高复杂度会议理解。

### 4.2 演示增强组合：可感知的 AI 设备

包含：

- #1 TTS 语音输出
- #5 VLM 场景理解实验版
- #8 手势识别陪伴交互 A 版

理由：

- 演示冲击力强，用户能直观看到“会看、会听、会说、能响应手势”。
- 适合答辩、路演、作品展示。

不建议作为默认生产组合：

- VLM 延迟和幻觉不适合接管实时控制。
- 手势误触发需要较多调参，因此 A 版只做陪伴交互，不控制云台。
- TTS 若默认外放，隐私和打扰问题明显。

### 4.3 会议增强组合：多人会议升级

包含：

- #1 TTS 可选播报
- #2 弱化版说话人分离
- #11 MQTT/HA 可选状态集成

理由：

- 直接增强现有会议录音、转写和摘要链路。
- 先用 DOA 区域或 Speaker A/B/C，避免过早承诺真实身份绑定。

不建议一开始部署完整声纹人脸绑定：

- 错标风险高。
- 隐私治理成本高。
- 模型部署和同步逻辑复杂。

## 5. 最终建议

短期最值得做的是：#3、#9、#10、#4。它们最贴近现有架构，投入小，能明显提升产品完整度。

中期可以做：#1、#7、#11。它们需要新增依赖或外部配置，但收益明确，边界也比较清楚。

应作为实验或暂缓的是：#5、#2、#6。#8 已收敛为 A 版低风险陪伴交互后，可以进入低-中难度批次；后续若要升级到拍照、控制云台或自动录音，再重新评估误触发、隐私和会话权限。尤其 #6 常驻语音助手和 #2 完整说话人绑定，不适合在主系统尚未完全稳定前直接并入生产链路。
